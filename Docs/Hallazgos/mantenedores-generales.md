> Ajuste de revisión incorporado después de detectar selects vacíos en GrupoVariedad/Variedad aunque existan Especies creadas.

| ID | Criterio | Aplicación |
|---|---|---|
| QAS-FK-01 | Todo formulario con FK debe probar carga real del catálogo padre con datos existentes. | No basta revisar que exista `useQuery` o que el build compile; se debe validar que el endpoint responda 200 y que el select muestre opciones. |
| QAS-FK-02 | Los parámetros usados por la UI deben cumplir el contrato del API. | Ej.: si API acepta `limit <= 100`, la UI no debe pedir `limit: 300` para poblar selects. |
| QAS-FK-03 | Los selects dependientes deben mostrar error visible o estado vacío explicativo cuando falla la carga. | Evitar que una consulta 422/500 se vea igual que "no hay datos". |
| QAS-FK-04 | Las cascadas deben probarse end-to-end. | Crear/verificar padre, abrir formulario hijo, seleccionar padre, verificar opciones dependientes filtradas y guardar. |
| QAS-FK-05 | Toda FK a otro mantenedor debe permitir crear el registro referenciado desde el mismo formulario. | Si Bodega depende de Comuna, el flujo debe permitir crear Comuna desde Bodega; si esa Comuna depende de Provincia/Región, el diseño debe resolver también esa cadena sin sacar al usuario del alta actual. |
